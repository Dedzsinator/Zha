The Zha system follows a sequential flow: Raw MIDI files are preprocessed into token sequences. A Markov chain generates an initial draft, the VAE transforms this draft via latent sampling, and the Transformer produces the final output, suitable for conversion back to MIDI.

\section{Data Flow and Model Integration}

\subsection{Sequential Pipeline Architecture}
A Zha rendszer három modelljét – Markov-lánc, VAE és Transformer – hierarchikus szerkezetben integrálja. Minden modell a zenei generálás egy specifikus aspektusát kezeli:

\begin{enumerate}
  \item \textbf{Markov-lánc (Strukturális alap)}: Akkordmenet generálása és harmonikus struktúra biztosítása. A detektált hangnemből kiindulva 8 akkordból álló progressziót alkot, amely harmonikus keretként szolgál.
  
  \item \textbf{VAE (Kreatív variáció)}: A Markov modell által meghatározott harmonikus struktúrán belül kreatív variációkat hoz létre. A latens térben történő mintavételezés lehetővé teszi a kreativitás mértékének szabályozását $\epsilon \sim \mathcal{N}(0, \text{creativity}^2)$ paraméterrel.
  
  \item \textbf{Transformer (Koherens finomítás)}: Long-range függőségeket és strukturális koherenciát biztosít. Multi-head attention mechanizmusával kapcsolatokat épít a távoli musical events között.
\end{enumerate}

\subsection{Combined Generation Algorithm}

Az integrált generálási folyamat a következő lépésekből áll:

\begin{algorithm}[ht]
\caption{Kombinált modell-generálás}
\begin{algorithmic}[1]
\Require Bemeneti MIDI fájl $M_{\text{input}}$, kreativitási paraméter $c \in [0,1]$
\Ensure Generált MIDI fájl $M_{\text{output}}$
\State $\mathbf{x}_{\text{input}} \gets \text{parse\_midi}(M_{\text{input}})$ \Comment{128-dimenziós feature vektor}
\State $(K, S) \gets \text{analyze\_key\_scale}(M_{\text{input}})$ \Comment{Hangnem és skála detektálás}
\State
\State \textbf{// Stage 1: Markov harmonikus struktúra}
\State $C \gets \text{markov.generate\_chord\_progression}(K, n=8)$ \Comment{8 akkord}
\State $\mathbf{m} \gets \text{markov.generate\_with\_chords}(K, C, \text{length}=64)$
\State
\State \textbf{// Stage 2: VAE kreatív variáció}
\State $(\mu, \log\sigma^2) \gets \text{VAE.encode}(\mathbf{x}_{\text{input}})$
\State $\mathbf{z} \sim \mathcal{N}(\mu, c \cdot \sigma^2)$ \Comment{Kreativitással skálázott sampling}
\State $\mathbf{v} \gets \text{VAE.decode}(\mathbf{z})$
\State $\mathbf{v}_{\text{filtered}} \gets \text{scale\_filter}(\mathbf{v}, S)$ \Comment{Skála szerinti szűrés}
\State
\State \textbf{// Stage 3: Transformer strukturális finomítás}
\State $\mathbf{t} \gets \text{Transformer}(\mathbf{x}_{\text{input}})$
\State $\mathbf{t}_{\text{range}} \gets \text{register\_limit}(\mathbf{t}, \text{octaves}=[3,6])$ \Comment{Hangterjedelem korlátozás}
\State
\State \textbf{// Stage 4: Weighted combination}
\State $\mathbf{y} \gets 0.5\mathbf{m} + 0.3\mathbf{v}_{\text{filtered}} + 0.2\mathbf{t}_{\text{range}}$
\State $\mathbf{y} \gets \max(0, \mathbf{y})$ \Comment{Non-negativitás biztosítása}
\State $\mathbf{y} \gets \mathbf{y} / \|\mathbf{y}\|_1$ \Comment{Normalizálás}
\State
\State $M_{\text{output}} \gets \text{notes\_to\_midi}(\mathbf{y}, C)$ \Comment{MIDI export akkordokkal}
\State \Return $M_{\text{output}}$
\end{algorithmic}
\end{algorithm}

\subsection{Scale Filtering and Register Constraints}

A VAE kimenet skála szerinti szűrése biztosítja, hogy a generált hangok harmonikusan illeszkedjenek a detektált hangnembe:

\[
v_{\text{filtered}}[i] = \begin{cases}
0.4 \cdot v[i] & \text{ha } (i \bmod 12) \in S \text{ és octave}(i) \in [3,6] \\
0.1 \cdot v[i] & \text{ha } (i \bmod 12) \notin S \\
0.3 \cdot v[i] & \text{ha octave}(i) \in \{2,7\} \text{ és } (i \bmod 12) \in S \\
0 & \text{egyébként}
\end{cases}
\]

ahol $S$ a detektált skála pitch class halmaza (pl. C-dúr esetén $S = \{0,2,4,5,7,9,11\}$), és $\text{octave}(i) = \lfloor i/12 \rfloor$.

A Transformer kimenet regiszter-korlátozott, hogy elkerülje a túl magas vagy mély hangokat:

\[
t_{\text{range}}[i] = \begin{cases}
0.3 \cdot t[i] & \text{ha octave}(i) \in [3,6] \\
0.1 \cdot t[i] & \text{egyébként}
\end{cases}
\]

\subsection{Data Flow Details}
\begin{enumerate}
  \item \textbf{Raw MIDI} — tárolva a \texttt{dataset/midi/} könyvtárban artist/album hierarchiában.
  \item \textbf{Processed Tokens} — kvantált és tokenizált JSON/NumPy tömbök a \texttt{dataset/processed/} mappában.
  \item \textbf{Markov Output} — akkordprogresszió és 128-dimenziós note probability distribution.
  \item \textbf{VAE Output} — latens térből dekódolt 128-dimenziós vektor, skálával szűrve.
  \item \textbf{Transformer Output} — koherens sequence prediction, regiszterrel limitálva.
  \item \textbf{Combined Output} — weighted combination, MIDI fájlba exportálva melody+chord trackekkel.
\end{enumerate}

\section{Data Preprocessing Pipeline}

\subsection{MIDI Parsing and Track Separation}
We parse each MIDI file using \texttt{music21} with intelligent track separation:
\begin{itemize}
  \item Discard tracks with fewer than 10 notes.
  \item Normalize tempo globally to 120 BPM to reduce variance.
  \item Remove non‐note events except sustain pedal controls.
  \item \textbf{Separate tracks into three categories:}
  \begin{itemize}
    \item \textbf{Melody}: Notes above bass range (MIDI $>$ 55), non-percussion
    \item \textbf{Bass}: Low notes (MIDI 21--55, approximately E0--G3)
    \item \textbf{Drums}: Percussion tracks (MIDI channel 10 or high-velocity short notes)
  \end{itemize}
\end{itemize}

\textbf{Track Separation Algorithm:}
\begin{enumerate}
  \item Parse multi-part MIDI scores with \texttt{music21.converter}
  \item Identify drum tracks by:
  \begin{itemize}
    \item MIDI channel 9 (standard percussion channel)
    \item Part names containing "drum", "percussion", or "perc"
    \item High velocity ($>90$) with short duration ($<0.5$ quarter notes)
  \end{itemize}
  \item Classify remaining notes by pitch:
  \begin{itemize}
    \item Bass range: MIDI notes $\le 55$ (typically played by bass instruments)
    \item Melody range: MIDI notes $> 55$ (main melodic content)
  \end{itemize}
  \item For chords, categorize by lowest pitch
  \item Sort all sequences chronologically by onset time
\end{enumerate}

This separation enables:
\begin{itemize}
  \item \textbf{Multi-track generation} in the Transformer model
  \item \textbf{Independent modeling} of melodic, harmonic, and rhythmic elements
  \item \textbf{Improved coherence} by avoiding mixed-range training data
  \item \textbf{Better bass line modeling} through dedicated low-frequency data
\end{itemize}

\subsection{Quantization}
Time is discretized to 16th‐note grid. For each note event:
\[
  t_{\text{quant}} = \mathrm{round}\!\bigl(t_{\text{actual}} \times 4 \bigr) \,/\,4.
\]
This ensures all onset and offset times snap to discrete bins.

\subsection{Tokenization Schema}
We adopt an extended REMI representation:
\begin{itemize}
  \item \texttt{NOTE\_ON\_p} / \texttt{NOTE\_OFF\_p} for pitches $p\in[0,127]$.
  \item \texttt{TIME\_SHIFT\_i} for $i\in[1,32]$ sixteenth‐note steps.
  \item \texttt{VELOCITY\_b} for eight velocity bins $b\in[1,8]$ computed via quantiles.
  \item \texttt{CONTROL\_SUSTAIN\_ON}, \texttt{CONTROL\_SUSTAIN\_OFF} tokens.
\end{itemize}
Total vocabulary size: 128 (notes) + 32 (time) + 8 (velocity) + 4 (controls) + 4 (special) = 176.

\section{Model Architectures}

\subsection{Markov Chain}
\subsubsection{Implementation Details}
\begin{itemize}
  \item Stored in \texttt{src/models/markov.py}.
  \item Supports $n$th‐order chains ($n\le4$).
  \item Uses Python \texttt{defaultdict(Counter)} for count accumulation.
  \item Applies Laplace smoothing with configurable $\alpha$.
\end{itemize}

\subsubsection{Sampling Algorithm}
\begin{enumerate}
  \item Initialize context with \texttt{START} tokens.
  \item For each step:
    \begin{itemize}
      \item Retrieve probability distribution $P(\cdot\mid\mathrm{context})$.
      \item Sample next token via cumulative multinomial sampling.
      \item Slide context window forward.
    \end{itemize}
  \item Terminate upon \texttt{END} token or length limit.
\end{enumerate}

\subsection{Variational Autoencoder}
\subsubsection{Encoder}
\[
  h_t = \mathrm{BiLSTM}(x_t, h_{t-1}),\quad
  \mu = W_\mu \, h_{T},\quad
  \log\sigma^2 = W_\sigma \, h_{T}.
\]
Implementation resides in \texttt{src/models/vae.py}.

\subsubsection{Decoder}
At each time step $t$:
\[
  \hat{x}_t = \mathrm{Softmax}\bigl(W_o [e_{t-1} \,\|\, z]\bigr),
\]
where $e_{t-1}$ is the embedding of the previous token and $z$ the sampled latent.

\subsubsection{Training Tricks}
\begin{itemize}
  \item \textbf{KL‐Annealing}: $\beta(e)=\min(1, e/E_{\mathrm{warmup}})$.
  \item \textbf{Free Bits}: floor KL per dimension at $0.1$ to prevent collapse.
  \item \textbf{Gradient Clipping}: $\lVert \nabla \rVert_\infty \le 1.0$.
\end{itemize}

\subsection{Transformer}
\subsubsection{Architecture}
\begin{itemize}
  \item Six layers, eight heads, $d_{\mathrm{model}}=512$.
  \item Rotary positional embeddings as per Su et al.\ (2020).
  \item Feed‐forward hidden size: 2048.
  \item Dropout: 0.1 on attention and feed‐forward sublayers.
\end{itemize}

\subsubsection{Relative Position Bias}
A learned bias matrix $B_{i,j}$ is added to attention logits to model musical interval relationships.

\section{Training Methodologies}

\subsection{Common Infrastructure}
\begin{itemize}
  \item All trainers in \texttt{src/trainers/}, share \texttt{TrainerBase} for logging and checkpointing.
  \item Use \texttt{Hydra} for hyperparameter configuration.
  \item Metrics written to TensorBoard and CSV.
\end{itemize}

\subsection{Markov Trainer}
\begin{itemize}
  \item Processes all token sequences, builds and saves $\text{P}_{\mathrm{matrix}}$.
  \item Validates by generating 100 sequences and checking unique transition coverage.
\end{itemize}

\subsection{VAE Trainer}
\begin{lstlisting}[language=Python]
for epoch in range(num_epochs):
    for batch in dataloader:
        recon, mu, logvar = model(batch)
        recon_loss = F.cross_entropy(recon, batch)
        kl = -0.5 * torch.sum(1 + logvar - mu.pow(2) - logvar.exp())
        loss = recon_loss + beta(epoch)*kl
        optimizer.zero_grad(); loss.backward()
        clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
\end{lstlisting}
Early stopping triggered if validation loss does not improve for 5 epochs.

\subsection{Transformer Trainer}
\begin{itemize}
  \item Mixed‐precision via \texttt{torch.cuda.amp}.
  \item Learning rate: warmup over first 5k steps, then cosine decay to zero by 200k steps.
  \item Gradient accumulation to simulate batch size of 512 when GPU memory limited.
\end{itemize}

\section{Inference Engine}

\subsection{End-to-End Script (\texttt{generate.py})}
\begin{lstlisting}[language=Python]
def generate(length=512, top_p=0.9, temp=1.0):
    # Stage 1: Markov
    mk = MarkovModel.load('checkpoints/markov.npy')
    seq1 = mk.sample(length=128)

    # Stage 2: VAE
    vae = VAE.load('checkpoints/vae.pt')
    z = vae.encode(seq1)
    seq2 = vae.decode(z, temperature=temp)

    # Stage 3: Transformer
    trans = MusicTransformer.load('checkpoints/trans.pt')
    seq3 = trans.generate(prefix=seq2, max_len=length, top_p=top_p)

    midi = detokenize(seq3)
    midi.write('output/generated.mid')
\end{lstlisting}

\subsection{Performance}
\begin{itemize}
  \item Markov sampling: $\approx$0.5 ms/token on CPU.
  \item VAE decode: $\approx$2 ms/token on GPU.
  \item Transformer generate: $\approx$4 ms/token with cache.
\end{itemize}

\[
HC = \frac{1}{N} \sum_{i=1}^{N} \mathbb{1}\left[ (p_i \bmod 12) \in S_{\text{key}} \right]

\section{Limitations and Future Work}

\subsection{Current Limitations}
\begin{itemize}
  \item The VAE occasionally experiences posterior collapse despite free‐bits regularization.
  \item Transformer stage incurs high memory and compute costs, limiting real‐time use on consumer hardware.
  \item Evaluation remains partly subjective; automatic metrics only approximate musical quality.
\end{itemize}
